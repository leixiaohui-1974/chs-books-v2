# 第4章 全栈架构：从 L0 到 L4

## 4.1 分层架构设计（L0核心算法、L1网关、L4智能体）

要把双容水箱的控制从“本地脚本”升级为“智能体可驱动”的水网应用，必须建立严谨的软硬件分层体系。本书采用的分层协议如下：

- **L0 核心算法层 (Core)**：包含物理方程 (`dynamics.py`) 和基础控制律 (`mpc.py`)，它只认纯数学输入，不关心外界如何调用。
- **L1 API 网关层 (Gateway)**：把 L0 的函数包装成网络接口或标准协议接口，实现机理与交互的解耦。
- **L2/L3 技能与编排层**：固定流程的控制任务流。
- **L4 认知智能体层 (Agent)**：基于大语言模型（LLM），能够通过自然语言理解意图，并向下组装 L1~L3 的能力。

## 4.2 基于 FastMCP 的模型封装：让大模型直接调用底层引擎

为了让 L4 的大模型能直接控制 L0 的物理世界，我们引入了 Model Context Protocol (MCP)。
在双容水箱场景下，我们通过 FastMCP 将底层的 `simulate` 或 `run_mpc` 封装为标准化 Tool。大模型无需知道 Python 细节，只需知道“我可以通过 `run_closed_loop` 传入设定值”。

这种封装是实现“自然语言到机器动作”转换的基础工程。

## 4.3 代码实战：`simulation_server.py` 的解耦设计

在 `core/coupled_tank/stack/l0_core.py` 到 `l1_gateway.py` 的代码演进中，我们展示了如何将业务逻辑与网络逻辑剥离。
L0 仅负责“返回积分后的水位列表”，而 L1 则将其包装为 JSON Schema，并添加参数校验、异常捕获。

## 4.4 代码实战：MPC 算法与 Agent 适配（Python实现）

当 L4 Agent 接收到诸如“将1号水箱水位稳定在 1.5 米”的指令时，`core/coupled_tank/stack/l4_agent_adapter.py` 会将这段意图解析为 `{ "target": 1.5 }`，传递给 L1，最终送达 L0 的 MPC 引擎。

对应测试 `tests/coupled_tank/test_ch04_stack_contract.py` 严格验证了“L0 -> L1 -> L4”这一调用链条的连通性与合法性，确保栈内通信坚不可摧。
