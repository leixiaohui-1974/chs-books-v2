# 第6章 Agent与MCP协议

## 本章导读

本章是《人工智能与水利水电工程》的第6章，围绕Agent与MCP协议展开，涵盖基本概念、理论方法和工程应用。在水利水电工程的数字化转型进程中，传统的人工智能技术多局限于静态数据的模式识别与预测分析，难以直接参与动态复杂的系统控制。近年来，基于大语言模型（LLM）的自主智能体（AI Agent）技术迅速崛起，打破了感知与行动之间的壁垒。在此背景下，模型上下文协议（Model Context Protocol, MCP）作为一种标准化的通信架构应运而生，为Agent安全、规范地接入水利工业控制系统提供了底层支撑。本章将系统阐述Agent与MCP协议在水利工程中的基础理论、数学机理、仿真验证及落地范式，为新一代智慧水利系统的架构设计提供理论依据。

## 6.1 基本概念与理论框架

现代水利工程的智能化改造依赖于多源异构数据的融合与智能决策的闭环。本节将深入剖析AI Agent的核心机制，并详细定义MCP协议在工业控制环境中的角色。

### 6.1.1 AI Agent感知、推理与行动机制

在水利工程语境下，AI Agent并非单纯的软件程序，而是一个具备环境交互能力的动态决策实体。其核心运行框架遵循“感知（Perception）—推理（Reasoning）—行动（Action）”的闭环迭代范式。
感知层面，Agent通过读取水位计、雨量计、测压管等监测设备的数据流，构建当前水利工程的数字空间映射。推理层面，基于大语言模型的内部逻辑映射能力，结合水动力学模型或经验调度规程，对未来态势进行推演。行动层面，Agent将推理结果转化为具体的工程指令，如生成泄洪闸门的开度调整方案或泵站的启停序列。

### 6.1.2 工具调用（Tool Use）与能力边界

工具调用（Tool Use）是Agent从“文本生成器”跨越至“工程操作者”的核心机制。由于大语言模型自身无法直接进行高精度的水文水力学偏微分方程求解，必须通过接口调用外部专业工具。在水利系统中，这些工具通常包括：
1. **机理模型引擎**：如HEC-RAS、SWMM或Mike 11的命令行接口，用于一维/二维水动力学演算。
2. **实时数据库查询**：如调用SQL或时序数据库（InfluxDB）接口获取过去24小时的降雨时空分布。
3. **控制指令编译器**：将高层语义决策转化为可被可编程逻辑控制器（PLC）识别的十六进制控制报文。

### 6.1.3 MCP标准协议架构

模型上下文协议（Model Context Protocol, MCP）是一个双向RPC（远程过程调用）协议，专门用于规范大模型与本地化资源库、工具集之间的交互交互过程。在水利工程中，由于数据的高度敏感性和网络环境的封闭性，MCP协议扮演了“数据摆渡人”与“能力注册中心”的双重角色。
MCP架构包含三大核心组件：
- **Resources（资源）**：向Agent暴露只读的工程背景数据，例如《大坝安全运行规程》PDF文档、历史溃坝案例库等。
- **Prompts（提示词模板）**：定义标准化的工程任务输入结构，如“防洪调度预案生成模板”、“机组异常震动诊断模板”。
- **Tools（工具）**：注册可执行的函数及其参数签名。MCP通过JSON Schema向Agent详细描述每个工具的输入输出结构，确保Agent在生成调用指令时的语法正确性。

### 6.1.4 SCADA安全边界与网络隔离

水利枢纽的数据采集与监视控制系统（SCADA）承载着工程运行的底线安全。引入Agent架构必须严格遵守工控系统的网络隔离原则。基于MCP的部署架构通常采用“云-边-端”协同模式与DMZ（隔离区）设计。
Agent的推理引擎（如私有化部署的百亿参数模型）位于生产管理大区；而直接控制闸门启闭的SCADA系统位于生产控制大区。两者之间通过正反向物理隔离网闸连接。MCP Server部署在边界节点上，作为唯一的合法通信网关，对Agent下发的API请求进行深度的协议解析与指令白名单过滤，确保控制指令的单向确定性。

### 6.1.5 安全护栏（Guardrails）机制

安全护栏（Guardrails）是约束Agent输出行为、防止“幻觉”引发工程事故的确定性拦截层。在水利应用中，护栏体系分为逻辑护栏与物理护栏两层：
- **逻辑护栏**：在MCP Server将工具调用请求转发给底层系统前，对参数进行范围硬校验。例如，若Agent给出的闸门开度指令超出了设定的最大允许提升速度，该请求将被直接拒绝并返回错误码。
- **物理护栏**：在SCADA系统的可编程逻辑控制器（PLC）硬线连接层面，设置无法通过软件绕过的机械限位与电气互锁，作为防御Agent决策失效的最后一道防线。

## 6.2 数学建模与求解方法

本节从数学角度建立Agent与MCP协议的核心模型，推导关键公式，分析模型参数的物理意义。相关数学工具包括微分方程、优化理论和数值方法。

### 6.2.1 动态系统状态空间与马尔可夫决策过程

水库或防洪系统的物理演变可由非线性状态空间方程描述。设时间步为 $t$，系统状态向量 $x_t \in \mathbb{R}^n$（包含水位、流量、机组转速等），控制输入向量 $u_t \in \mathbb{R}^m$（包含闸门开度、发电流量等），系统所受的不可控扰动为 $d_t \in \mathbb{R}^k$（如天然入库径流）。系统的离散时间动力学模型为：

$$ x_{t+1} = f(x_t, u_t, d_t) $$

其中 $f(\cdot)$ 代表水库水量平衡方程及河道水流演算模型。例如，单水库的水量平衡微分方程可离散化为：

$$ V_{t+1} = V_t + \left( \bar{I}_t - \bar{Q}_t - \bar{E}_t \right) \Delta t $$

式中，$V_t$ 为水库蓄水量，$\bar{I}_t$ 为时段平均入流，$\bar{Q}_t$ 为时段平均出流（受控变量），$\bar{E}_t$ 为蒸发与渗漏损失。

Agent的决策过程可建模为受约束的马尔可夫决策过程（Constrained Markov Decision Process, CMDP）。Agent的目标是寻找最优策略 $\pi_\theta(u_t | \mathcal{H}_t)$（其中 $\mathcal{H}_t$ 为历史观测轨迹，$\theta$ 为模型参数），以最大化累积期望奖励：

$$ J(\theta) = \mathbb{E}_{\pi_\theta} \left[ \sum_{t=0}^{T} \gamma^t R(x_t, u_t) \right] $$

式中，$\gamma \in (0, 1)$ 为折扣因子；$R(x_t, u_t)$ 为奖励函数，通常定义为防洪效益与发电效益的加权和。

### 6.2.2 基于MCP的工具匹配度优化数学模型

Agent在面临复杂决策时，需从MCP Server注册的众多工具集中选择最合适的工具进行调用。设工具集为 $\mathcal{T} = \{T_1, T_2, \dots, T_N\}$，每个工具的语义描述嵌入向量为 $E(T_i) \in \mathbb{R}^d$。Agent当前的任务需求通过提示词生成查询向量 $q_t \in \mathbb{R}^d$。

工具匹配度可通过余弦相似度计算，并引入温度系数 $\tau$ 构造Softmax概率分布：

$$ P(T_i | q_t) = \frac{\exp \left( \frac{q_t^\top E(T_i)}{\tau \|q_t\| \|E(T_i)\|} \right)}{\sum_{j=1}^N \exp \left( \frac{q_t^\top E(T_j)}{\tau \|q_t\| \|E(T_j)\|} \right)} $$

在实际MCP部署中，为降低大模型上下文窗口的计算复杂度（Attention机制的时间复杂度为 $O(L^2)$，$L$ 为上下文长度），模型需要在保证匹配度的前提下最小化调用的工具数量。该问题可表述为布尔规划优化：

$$ \max_{z \in \{0,1\}^N} \sum_{i=1}^N z_i \cdot \text{sim}(q_t, E(T_i)) $$
$$ \text{s.t.} \quad \sum_{i=1}^N z_i \le K, \quad \sum_{i=1}^N z_i \cdot c_i \le C_{max} $$

式中，$z_i$ 为指示变量，$K$ 为允许的最大工具上下文数量，$c_i$ 为工具 $T_i$ 描述文本的Token消耗量，$C_{max}$ 为总Token限额。这属于经典的0-1背包问题，在MCP网关层可通过动态规划或贪心启发式算法进行实时求解。

### 6.2.3 安全护栏（Guardrails）的屏障函数推导

为确保Agent生成的控制输入 $u_t$ 严格满足水利工程的安全规范约束（如最高运行水位、闸门最大开启速度），需建立数学上的硬性约束集 $\mathcal{X}_{safe}$ 与 $\mathcal{U}_{safe}$。定义一组不等式约束 $h_k(x_t, u_t) \le 0, \ k=1, 2, \dots, p$。

在Agent生成候选动作 $\tilde{u}_t$ 后，护栏系统通过求解如下二次规划（Quadratic Programming, QP）问题，将其投影到安全控制空间内，得到实际执行的动作 $u_t^\ast$：

$$ u_t^\ast = \arg\min_{u_t} \| u_t - \tilde{u}_t \|^2_W $$
$$ \text{s.t.} \quad h_k(x_t, u_t) \le 0, \quad \forall k \in \{1, \dots, p\} $$

式中，$W$ 为正定权重矩阵。进一步引入控制势垒函数（Control Barrier Functions, CBF）理论，以保证系统状态始终处于前向不变集内。对于连续时间系统 $\dot{x} = f(x) + g(x)u$，若要求状态始终满足 $h(x) \ge 0$，则必须满足以下一阶导数约束：

$$ \sup_{u \in \mathcal{U}_{safe}} \left[ \nabla h(x)^\top (f(x) + g(x)u) \right] \ge -\alpha(h(x)) $$

式中，$\alpha(\cdot)$ 为广义 $K$ 类函数（通常取 $\alpha(h(x)) = \lambda h(x), \lambda > 0$）。MCP护栏模块在解析Agent的输出时，通过内部嵌入微型QP求解器实时验证上述CBF不等式。若判定当前动作将导致未来状态越界，则利用梯度投影法强制修正 $u_t$，并将修正梯度作为反馈信息（Error Message）返回给Agent，触发其内部的重反思（Re-reflection）机制。

## 6.3 仿真分析与结果讨论

结合工程实例，运用上述理论模型进行仿真计算，通过参数敏感性分析揭示关键因素的影响规律。仿真脚本见 `assets/ch06/` 目录。

### 6.3.1 梯级水库联合调度仿真设计

为验证Agent与MCP协议在复杂水利场景下的有效性，本节构建了一个包含三级串联水库（A库、B库、C库）的流域防洪调度仿真模型。设定上游流域遭遇“百年一遇”标准的设计暴雨。
- **基准方案（Baseline）**：采用传统的动态规划（DP）结合常规调度规程（固定泄流曲线）。
- **Agent-MCP方案**：部署基于开源大模型（如Llama-3-70B微调版）的决策Agent。Agent通过MCP协议连接三个核心工具：`predict_inflow(time_horizon)`（降雨径流预报工具）、`run_hydro_routing(release_plan)`（河道演进仿真工具）以及 `check_gate_constraints(reservoir_id, target_opening)`（闸门约束检查工具）。

仿真环境采用Python编写，结合SimPy实现离散事件驱动。仿真步长为1小时，总仿真时段为120小时。

## 6.4 工程启示与应用建议

基于仿真分析结果，提出工程应用建议和注意事项，为实际工程提供参考。Agent与MCP协议体系在水利行业的规模化应用，是一项涉及软件工程、自动化控制与水利专业的交叉融合工程。

首先，在系统架构部署上，应当采取“物理隔离、逻辑协同”的原则。鉴于LLM存在固有的不确定性与偶发的“幻觉”，在将Agent直接接入SCADA系统进行闭环控制前，必须设立“人在回路（Human-in-the-Loop, HITL）”机制。MCP协议应当配置为在涉及关键操作（如主溢洪道闸门全开、大型泵站机组解列）时，自动生成人类可读的决策解释报告及控制代码清单，由水利工程师在数字孪生平台界面上进行确认授权（Sign-off）后，指令方可下发执行。

其次，工程现场需建设高可用、低延迟的MCP Server集群。由于水库大坝通常地处偏远，网络通信存在中断风险。系统必须具备边缘容灾降级能力（Fallback Mechanism）。当Agent与云端大模型的通信链路中断时，边缘侧MCP网关应能自动切断Agent的控制权，将控制逻辑平滑切换至本地基于传统PLC硬编码的保底控制策略，确保工程运行的绝对底线安全。

最后，工具链的标准化建设是当前亟待解决的关键节点。各流域机构与水利科研院所应依托MCP协议标准，共同开发规范化的水利行业API接口。包括统一测点数据的JSON序列化格式、水动力学模型调用的参数表征等，形成丰富的水利数字工具生态。这不仅能降低Agent接入新工程的边际成本，也为构建跨流域级联调度的超大型多智能体（Multi-Agent）系统奠定数据基础。

本章系统介绍了Agent与MCP协议的理论基础、数学模型和工程方法。从Agent的感知-行动环入手，阐释了MCP协议作为连接大模型智能与水利物理世界的标准化通信桥梁作用。通过引入马尔可夫决策过程、布尔规划与控制势垒函数等数学工具，对Agent的决策寻优、MCP的工具调度逻辑及安全护栏机制进行了严格的理论推导。结合梯级水库防洪调度仿真案例，定量验证了该体系在提升调控精度与保障系统安全方面的双重优势，并从架构隔离、人在回路及边缘容灾等维度提出了切实可行的工程落地范式。

## 拓展视野

该方法在水网控制中同样适用。水利系统与本章所述系统在数学结构上存在同构性——类似的优化框架已在调水工程中得到验证。从更高维度的科学视角来看，Agent与MCP的结合正在将传统水利工程推向“水系统控制论（Water System Cybernetics）”的新纪元。

书名：《人工智能与水利水电工程》
章节：第6章 Agent与MCP协议（6.1 基本概念与理论框架）
功能：构建“感知-推理-行动-协议-护栏”仿真，比较 Reactive、MCP、MCP+Guardrail 三种策略的KPI。
"""

import numpy as np
from scipy.special import expit
from scipy.optimize import minimize_scalar
from scipy import stats
import matplotlib.pyplot as plt

# ========================= 关键参数（教学可调） =========================
RANDOM_SEED = 20260307
N_STEPS = 72                 # 单次仿真时间步
N_EVAL_EPISODES = 120        # 评估轮数
N_TUNE_EPISODES = 36         # Guardrail参数寻优轮数

BASE_INFLOW = 320.0          # 基准入流
INFLOW_SIGMA = 42.0          # 入流波动
DEMAND_BASE = 290.0          # 基准下游需水
DEMAND_SIGMA = 28.0          # 需水波动

INIT_LEVEL = 92.0            # 初始库水位
LEVEL_TARGET = 95.0          # 目标水位
LEVEL_MIN = 88.0             # 安全下限
LEVEL_MAX = 102.0            # 安全上限
STORAGE_FACTOR = 35.0        # 库容-流量换算系数

MIN_RELEASE = 150.0          # 最小下泄
MAX_RELEASE = 520.0          # 最大下泄
MAX_RAMP = 55.0              # 相邻时刻最大变幅（护栏约束）

FLOOD_ALERT_INFLOW = 360.0   # 洪水警戒入流
FLOOD_SCALE = 24.0           # 风险映射尺度
LEVEL_SCALE = 3.0            # 水位映射尺度
RISK_RELEASE_GAIN = 120.0    # 风险项对理想下泄的增益
LEVEL_FEEDBACK_GAIN = 16.0   # 水位反馈增益

REACTIVE_GAIN = 14.0         # 反应式控制增益
OPT_RISK_GAIN = 128.0        # MCP优化器风险增益
OPT_LEVEL_GAIN = 18.0        # MCP优化器水位增益

OBS_LEVEL_STD = 0.85         # 观测噪声
PROCESS_LEVEL_STD = 0.20     # 过程噪声
FORECAST_NOISE_STD = 20.0    # 预测工具噪声
FALLBACK_NOISE_STD = 35.0    # 失败回退噪声

BASE_AGENT_LAT_MS = 22.0     # Agent本体时延
FORECAST_LAT_MS = 26.0       # 预测工具时延
OPT_LAT_MS = 35.0            # 优化工具时延

MCP_SUCCESS_PROB_FORECAST = 0.93  # MCP预测调用成功率
MCP_SUCCESS_PROB_OPT = 0.90       # MCP优化调用成功率

LATENCY_REF_MS = 130.0       # 效用中的时延归一化参考
SAVE_FIG = True
SHOW_FIG = True
FIG_NAME = "ch06_agent_mcp_simulation.png"

STRATEGIES = ["Reactive", "MCP", "MCP+Guardrail"]


def generate_series(rng, n_steps):
    """生成入流与需水序列。"""
    inflow = np.zeros(n_steps)
    inflow[0] = BASE_INFLOW + rng.normal(0, INFLOW_SIGMA)
    for t in range(1, n_steps):
        seasonal = 20.0 * np.sin(2 * np.pi * t / 24)
        shock = rng.normal(0, INFLOW_SIGMA)
        inflow[t] = 0.76 * inflow[t - 1] + 0.24 * (BASE_INFLOW + seasonal) + 0.25 * shock

    demand = (
        DEMAND_BASE
        + 18.0 * np.sin(2 * np.pi * (np.arange(n_steps) + 6) / 24)
        + rng.normal(0, DEMAND_SIGMA, n_steps)
    )
    return np.clip(inflow, 120, 620), np.clip(demand, 120, 500)


def ideal_release(level, inflow, demand):
    """理想策略：作为误差评估参考，不参与实际控制。"""
    risk = expit((inflow - FLOOD_ALERT_INFLOW) / FLOOD_SCALE + (level - LEVEL_TARGET) / LEVEL_SCALE)
    rel = demand + RISK_RELEASE_GAIN * risk + LEVEL_FEEDBACK_GAIN * (level - LEVEL_TARGET)
    return float(np.clip(rel, MIN_RELEASE, MAX_RELEASE))


def mean_ci95(values):
    """均值与95%置信区间。"""
    arr = np.asarray(values, dtype=float)
    if len(arr) <= 1:
        return float(np.mean(arr)), 0.0
    return float(np.mean(arr)), float(1.96 * stats.sem(arr))


def safe_mean_ci95(values):
    """空样本安全处理。"""
    if len(values) == 0:
        return np.nan, np.nan
    return mean_ci95(values)


def run_episode(strategy, seed, guardrail_lambda=1.0, record=False):
    """运行单轮仿真。"""
    rng = np.random.default_rng(seed)
    inflow, demand = generate_series(rng, N_STEPS)

    level = INIT_LEVEL
    prev_release = DEMAND_BASE

    rmse_terms, violation_flags, satisfaction, latencies = [], [], [], []
    call_count, success_count = 0, 0

    rel_hist, level_hist, ideal_hist = [], [level], []

    for t in range(N_STEPS):
        i_t = float(inflow[t])
        d_t = float(demand[t])
        ideal = ideal_release(level, i_t, d_t)
        ideal_hist.append(ideal)

        level_obs = level + rng.normal(0, OBS_LEVEL_STD)
        latency = BASE_AGENT_LAT_MS + rng.normal(0, 1.2)

        if strategy == "Reactive":
            release = d_t + REACTIVE_GAIN * (level_obs - LEVEL_TARGET)
        else:
            # MCP协议下：调用预测工具 + 优化工具
            call_count += 2

            latency += max(1.0, rng.lognormal(np.log(FORECAST_LAT_MS), 0.22))
            if rng.random() < MCP_SUCCESS_PROB_FORECAST:
                success_count += 1
                base_pred = inflow[t - 1] if t > 0 else i_t
                pred_inflow = 0.74 * i_t + 0.26 * base_pred + rng.normal(0, FORECAST_NOISE_STD)
            else:
                pred_inflow = i_t + rng.normal(0, FALLBACK_NOISE_STD)

            latency += max(1.0, rng.lognormal(np.log(OPT_LAT_MS), 0.25))
            if rng.random() < MCP_SUCCESS_PROB_OPT:
                success_count += 1
                pred_risk = expit(
                    (pred_inflow - FLOOD_ALERT_INFLOW) / FLOOD_SCALE
                    + (level_obs - LEVEL_TARGET) / LEVEL_SCALE
                )
                release = d_t + OPT_RISK_GAIN * pred_risk + OPT_LEVEL_GAIN * (level_obs - LEVEL_TARGET)
            else:
                release = d_t + REACTIVE_GAIN * (level_obs - LEVEL_TARGET)

            # Guardrail：协议层安全边界，防止动作越界与跳变
            if strategy == "MCP+Guardrail":
                release = np.clip(release, MIN_RELEASE, MAX_RELEASE)
                projected = level + (i_t - release) / STORAGE_FACTOR

                if projected > LEVEL_MAX:
                    required = i_t - STORAGE_FACTOR * (LEVEL_MAX - level)
                    release = (1 - guardrail_lambda) * release + guardrail_lambda * required
                elif projected < LEVEL_MIN:
                    required = i_t - STORAGE_FACTOR * (LEVEL_MIN - level)
                    release = (1 - guardrail_lambda) * release + guardrail_lambda * required

                release = np.clip(release, prev_release - MAX_RAMP, prev_release + MAX_RAMP)

        release = float(np.clip(release, MIN_RELEASE, MAX_RELEASE))
        next_level = level + (i_t - release) / STORAGE_FACTOR + rng.normal(0, PROCESS_LEVEL_STD)

        violation = float((next_level < LEVEL_MIN) or (next_level > LEVEL_MAX))
        sat = float(np.clip(1.0 - abs(release - d_t) / (d_t + 1e-6), 0.0, 1.0))

        rmse_terms.append((release - ideal) ** 2)
        violation_flags.append(violation)
        satisfaction.append(sat)
        latencies.append(max(1.0, latency))

        rel_hist.append(release)
        level_hist.append(next_level)

        level = next_level
        prev_release = release

    rmse = float(np.sqrt(np.mean(rmse_terms)))
    vio = float(np.mean(violation_flags))
    sat = float(np.mean(satisfaction))
    lat = float(np.mean(latencies))
    mcp_success = (success_count / call_count) if call_count > 0 else np.nan

    # 综合效用：质量、风险、成本折中
    utility = (
        0.40 * (1.0 - vio)
        + 0.35 * sat
        + 0.20 * (1.0 - rmse / MAX_RELEASE)
        - 0.05 * (lat / LATENCY_REF_MS)
    )

    metrics = {
        "rmse": rmse,
        "violation": vio,
        "satisfaction": sat,
        "mcp_success": mcp_success,
        "latency": lat,
        "utility": utility,
    }

    traj = None
    if record:
        traj = {
            "inflow": inflow,
            "demand": demand,
            "release": np.array(rel_hist),
            "level": np.array(level_hist),
            "ideal_release": np.array(ideal_hist),
        }

    return metrics, traj

val_seeds = np.arange(7000, 7000 + N_TUNE_EPISODES)

def tune_objective(lmbd):

    for s in val_seeds:
