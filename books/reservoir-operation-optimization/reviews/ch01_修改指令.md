## 修改1: 逆时序计算公式 (line 70)

### 当前错误
```markdown
V_{t-1} = V_t - (Q_{in,t} - Q_{out,t} - E_t) \cdot \Delta t
```

### 修改为
```markdown
V_{t-1} = V_t + (Q_{out,t} - Q_{in,t} + E_t) \cdot \Delta t
```

### 修改说明
在逆时序递推中,从t时刻推算t-1时刻的库容时:
- t-1时刻的水量 = t时刻的水量 + 这个时段流出的水量 - 这个时段流入的水量 + 蒸发损失
- 符号规则: 流出为正(增加前一时刻库容),流入为负(减少前一时刻库容)

### 补充解释 (可选,插入line 70后)
```markdown
**说明**: 逆时序递推时,时间箭头反向,因此水量变化的符号也需反向。从物理意义理解:
- 若t时刻库容为V_t,要达到这个状态,t-1时刻的库容必须满足:
  - V_{t-1} + (流入 - 流出 - 蒸发) × Δt = V_t
  - 即 V_{t-1} = V_t - (流入 - 流出 - 蒸发) × Δt = V_t + (流出 - 流入 + 蒸发) × Δt
```

## 修改2: 包络线方向描述 (line 71)

### 当前错误
```markdown
从终点向起点逐步收缩可行域
```

### 修改为
```markdown
从终点向起点逐步构建可行域包络线。每个时段的可行域由下一时段所有可行状态的可达前驱集合决定,因此可行域边界呈现"漏斗状"扩展:越靠近起点,可行域越大(因为有更多路径可以到达终点的可行状态)。
```

### 修改说明
逆时序DP的可行域构建逻辑:
1. 终点(t=T): 可行域最小,仅为终端约束V_T ∈ [V_target - ε, V_target + ε]
2. 倒数第二步(t=T-1): 可行域为"所有能在一步内到达终点可行域的状态",通常比终点大
3. 依此类推,越往前推,可行域越大
4. 起点(t=0): 可行域最大,但必须包含初始状态V_0

这与"收缩"的描述相反,应该是"扩展"或"构建包络线"。

### 可视化示意 (可选,插入line 71后)
```markdown
**可行域演化示意**:
```
t=0        t=1        t=2      ...      t=T-1      t=T
█████      ████       ███                ██         █
█████  →   ████   →   ███   →   ...  →  ██    →    █  (终点)
█████      ████       ███                ██         █
(起点)    (扩展)     (扩展)            (收窄)    (最窄)
```
逆时序递推方向: ← ← ← ← ←
可行域变化: 从终点的窄域逐步扩展到起点的宽域

## 修改3: 动态防洪限制水位合规性 (line 161)

### 当前不足
```markdown
动态防洪限制水位可根据预报信息实时调整
```

### 修改为
```markdown
动态防洪限制水位可根据预报信息实时调整,但需满足以下条件:

1. **预报精度要求**: 预报合格率≥85%,预见期≥3天(《水库调度规程编制导则》SL706-2015第6.3.4条)
2. **审批流程**:
   - 一类水库(大型及以上): 调整方案需报流域管理机构审批
   - 二类水库(中型): 报省级水行政主管部门备案
   - 三类水库(小型): 报县级水行政主管部门备案
3. **安全约束**:
   - 确保下游防洪标准不降低
   - 动态限制水位不得高于设计洪水位
   - 预报失准时能在6小时内恢复到设计防洪限制水位
4. **应急预案**: 建立"预报-调度-监测-反馈"闭环,实时监测预报误差,超过阈值立即启动应急泄洪

**工程案例**: 三峡水库自2008年起实施动态防洪限制水位调度,汛前(6-7月)限制水位145m,汛末(8-9月)根据预报逐步抬升至155-160m,累计增加发电量约30亿kWh/年,但需满足长江中下游防洪安全约束(荆江河段不超过保证水位45.0m)。
```

### 修改说明
补充了四个关键维度:
1. 技术条件(预报精度)
2. 管理流程(审批备案)
3. 安全约束(硬性边界)
4. 应急机制(风险管控)

这些内容确保读者理解"动态调整"不是任意调整,而是在严格约束下的优化。
