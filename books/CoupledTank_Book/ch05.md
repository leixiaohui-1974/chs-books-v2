# 第5章 CC-Desktop 工作台与智能决策助理

## 5.1 统一交互界面：告别黑框框，拥抱大模型

一直以来，工业控制系统和仿真软件的界面都是“工程师专属”的：密密麻麻的参数框、晦涩的脚本命令行。而在 HydroDesktop（原 CC-Desktop）工作台中，我们引入了以对话交互为核心的大模型助理界面。
这意味着，操作员不再需要记住“改 PID 积分时间要点哪个菜单”，而是直接输入：“一号水箱液位波动有点大，帮我看看怎么回事并稳定它”。

## 5.2 意图识别：将自然语言转化为控制参数

大模型助理的第一步工作是“意图转译（NLU Intent）”。
在 `core/coupled_tank/nlu_intent.py` 中，我们定义了转译逻辑的基线契约：当用户说“稳定在1.5米”时，AI 必须能将其准确提取为 `{ "target": 1.5 }` 这样的结构化 API 参数，并将其传递给下层的 MPC 或 PID 控制器。
这种转译过程需要考虑：
1. 角色的越权识别（操作员能否改目标值？）。
2. 参数的合理性校验（1.5米是否超出了 2.0米的量程上限？）。

## 5.3 JSON 标准模板机制（MBD映射）：蓄水节点与控制设备的通信协议

当下层模型计算出结果后（比如未来 10 分钟的水位预测曲线），大模型必须负责把这些干瘪的数字变成人话。
为了避免大模型“胡编乱造”，我们在 `core/coupled_tank/schema_template.py` 中强制规定了返回数据的“结构化降维模板”。
所有物理节点必须映射为特定的 MBD（Model-Based Design）对象类型，比如：
- `storage_node`（蓄水节点：记录水位、容积、溢流风险）
- `control_equipment`（控制设备：记录阀门开度、饱和状态）

对应 `tests/coupled_tank/test_ch05_intent_to_json.py` 严格测试了这种正向解析和反向包装的结构稳定性，确保 AI 不会偏离物理契约。

## 本章小结与证据锚点

- **核心结论**：智能助理的实质是“自然语言与结构化控制契约之间的稳定翻译器”。
- **代码锚点**：`core/coupled_tank/nlu_intent.py`, `core/coupled_tank/schema_template.py`
- **测试证据**：`tests/coupled_tank/test_ch05_intent_to_json.py`
