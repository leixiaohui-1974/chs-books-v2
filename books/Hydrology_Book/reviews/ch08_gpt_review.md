## 1. 章节概述

本章介绍MCP（Model Context Protocol）作为大语言模型与水文物理系统之间的"安全走廊"，系统阐述了Tools/Resources/Prompts三大支柱。章节的核心贡献在于将LLM幻觉（Hallucination）问题的工业控制风险量化，并通过强类型Schema约束和物理可行域限制等机制提出防范方案。将AI认知层与水文物理层的接口标准化是领域内较为前沿的架构设计。

## 3. P0 问题列表（必须修改）

**P0-1【MCP协议描述需标注版本和来源】**
- 位置：第2.1节，MCP三大支柱描述
- 问题：MCP（Model Context Protocol）是Anthropic于2024年底发布的开放协议，具有版本迭代特性。正文对MCP的描述未注明协议版本，若协议后续迭代，内容可能过时。
- 修复：在首次提及MCP时注明"MCP（Model Context Protocol，v1.0，Anthropic 2024）"，并在参考文献中引用官方规范文档URL。

**P0-2【LLM幻觉对工业控制的风险需量化警示等级】**
- 位置：第2.2节，幻觉风险描述
- 问题：正文描述"幻觉可能比洪水更致命"具有警示价值，但缺乏具体的风险等级框架（如国标安全完整性等级SIL），水务控制系统通常需要达到SIL-2或SIL-3标准，LLM集成的风险需要系统性评估。
- 修复：增加一段说明"当LLM作为决策辅助工具时，建议遵循IEC 61508安全完整性等级（SIL）框架，对LLM输出的可靠性进行独立验证，不应将LLM直接接入闸门等安全关键系统的控制回路"。

## 4. P1 问题列表（强烈建议修改）

**P1-1【Schema约束的物理可行域映射示例过于简单】**
- 位置：第2.1节，Tools描述中的函数签名示例
- 问题：示例函数`run_forecast(basin_id, rain_mm, duration_h)`仅展示了简单参数，未展示Schema约束的具体格式（如JSON Schema中的`minimum`、`maximum`、`enum`等约束字段），读者无法直接实现。
- 建议：增加一个JSON Schema示例，展示`rain_mm`参数的物理约束：`{"type":"number","minimum":0,"maximum":2000,"description":"累计降雨量(mm)，不超过历史极值2000mm"}`。

**P1-2【与第9章Skill的边界未明确】**
- 位置：第2.1节，Prompts描述
- 问题：MCP的Prompts与第9章介绍的Skill（标准操作程序SOP）在功能上有重叠，两者的分工未在本章明确，读者可能混淆"System Prompt约束LLM行为"与"Skill编码化SOP"的区别。
- 建议：在本章末尾增加"MCP vs Skill的分工：MCP定义LLM与外部系统的接口协议，Skill定义LLM内部的思考程序；MCP解决'怎么调用'，Skill解决'调用什么、按什么顺序'"。

**P1-3【"认知层"与"物理层"术语需在CHS框架中对齐】**
- 位置：第2.1节开头
- 问题：正文引入"认知层"概念，但CHS六元组（P/A/S/D/C/O）中没有明确对应"认知层"的元素，与第4章MBD体系的接口不清晰。
- 建议：明确说明"认知层对应CHS中的C（控制器）元素的高级功能扩展，MCP是C元素与外部LLM能力集成的标准化协议"。

## 5. P2 问题列表（酌情考虑）

**P2-1【"幻觉"术语使用建议增加英文对照】**
- 位置：第2.2节标题
- 问题：LLM"幻觉"对应英文Hallucination，建议首次出现时标注，便于读者查阅国际文献。

**P2-2【案例场景中闸门开度"0-1"量纲应更明确】**
- 位置：第2.1节，System Prompt示例中"闸门开度范围0-1"
- 问题：闸门开度0-1（无量纲比例）是工程惯例，但实际工程中有时用百分比（0-100%）或实际开度（cm），建议在约束声明中明确单位。

本章是全书最前沿的内容之一，将MCP协议引入水文领域具有较强的探索价值。由于MCP协议本身较新（2024年），文献支撑相对薄弱，需要通过引用官方文档（P0-1）和增加工业安全等级框架（P0-2）来提升可信度。总体而言，本章内容完整度较高，是教材的创新亮点章节。
