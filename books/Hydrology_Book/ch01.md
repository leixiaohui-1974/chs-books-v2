# 第1章 2026年的智能水文为什么重要

## 1.1 城市内涝、干旱压力与基础设施老化的挑战

2026年的水问题不再是单一灾种问题，而是“极端降雨-持续干旱-设施老化”叠加下的复合风险问题。
同一座城市可能在汛期承受短历时强降雨导致的内涝冲击，在枯季又面临来水不足、地下水超采、生态基流受挤压的压力。与此同时，大量管网、泵闸、监测站点进入老化期，系统韧性下降，故障传播速度加快。

对管理者而言，关键不只是“有没有水”，而是：

1. 水在什么时间、什么空间、以什么强度到达。
2. 设施在这些工况下还能否安全、稳定运行。
3. 风险是否能在灾前被量化并被提前干预。

这也是第1章映射到 `core/hydrology/domain_context.py` 的根本原因。该模块的核心逻辑不是“写背景介绍”，而是把背景转为结构化上下文：
把气候压力、城市暴露度、资产状态、业务目标、约束边界组织成可计算的“领域上下文对象（Domain Context）”，为后续预报、调度和告警提供统一输入。

从工程表达上，可以把风险理解为：
$$ % (Eq. Auto-Aligned)Risk = Hazard \times Exposure \times Vulnerability$$
其中 `domain_context.py` 的作用，就是明确 Hazard/Exposure/Vulnerability 在水文业务中的可落地变量与取值规则，而不是停留在口号层面。

## 1.2 传统被动响应模式向预测性调控的业务转变

传统模式以“事件发生后处置”为主，典型流程是：
报警出现 -> 人工研判 -> 下达调度命令 -> 现场反馈。
这种模式在常规工况可用，但在高频极端事件下会暴露三类短板：

1. 时间滞后：关键窗口期往往早于人工决策周期。
2. 信息割裂：雨情、水情、工情、险情分散在不同系统。
3. 策略不连续：跨班组、跨部门、跨流域联动困难。

预测性调控强调“前置感知-前置研判-前置动作”，其业务闭环是：

1. 监测：融合站点、雷达、遥感和历史库。
2. 预报：给出未来时段的来水、水位、风险概率。
3. 决策：在约束条件下选择可执行方案。
4. 执行：泵闸库网协同动作。
5. 复盘：用结果反校模型与策略。

`core/hydrology/domain_context.py` 在这一转变中的核心逻辑可归纳为三点：

1. 场景分层：平时、临战、应急、恢复，不同场景绑定不同决策优先级。
2. 目标分层：防洪安全、供水保障、生态约束、能耗成本等目标可并行表达。
3. 约束分层：法规红线、设备极限、调度边界以机器可读方式显式化。

因此，第1章不是“讲趋势”，而是定义“为什么必须从反应式转向预测式”的可验证前提。
对应通过证据应指向 `tests/hydrology/test_ch01_domain_context.py`（等价映射：`tests/hydrology/test_ch01_context_assumptions.py`），用于验证上下文假设、场景边界和约束表达是否一致、可复现。

## 1.3 什么是 Hydrology Intelligence（水文智能）

Hydrology Intelligence 不是单一算法，也不是把大模型接进系统就算“智能化”。
它是“物理机理 + 数据融合 + 决策执行 + 证据闭环”的工程体系。

可以把它分为四层：

1. 机理层：产汇流、河道汇流、地下水-地表水交换等过程模型。
2. 数据层：多源异构数据统一语义、统一时空索引、统一质量标识。
3. 决策层：预报、优化、告警、应急策略联动。
4. 证据层：每个结论都能追溯到代码、测试和运行结果。

在映射矩阵中，Ch1 对应 `hydrology_server.describe_context`，本质就是把 `domain_context.py` 的上下文逻辑服务化，让“背景认知”变成“可调用能力”。
这一步让业务人员、算法人员和运维人员在同一上下文下协作，避免“各说各话”。

## 本章小结

第1章回答了“为什么智能水文在2026年是刚需”：

1. 风险形态已从单灾种变为复合风险。
2. 被动处置在时效和协同上存在结构性上限。
3. 领域上下文必须代码化、服务化、测试化。
4. `domain_context.py` 与 Ch1 测试共同构成“背景即能力”的证据起点。
