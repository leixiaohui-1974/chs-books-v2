# 第10章 Agent 在水务运营中的架构

## 10.1 调度规划 Agent、数据分析 Agent、控制 Agent 的分工

一个复杂的水网系统不能由一个单一的全能大模型来接管。微服务架构的思想同样适用于智能体（Agent）。
在水文智能体系中，我们把大模型切分为多个具有特定领域职责的子智能体：
- **调度规划 Agent (`dispatcher.py`)**：负责根据气象预报和水库水位，生成未来24小时的放水计划草案。
- **数据分析 Agent (`analyzer.py`)**：负责紧盯传感器的数据流，一旦出现流量异常或水质超标，立即启动归因分析并排查故障源头。
- **控制 Agent (`controller.py`)**：负责接收规划指令，把它们转译为对物理泵站和闸门的开度调节指令。
这种分工机制记录在 `core/hydrology/agents/` 目录中。

## 10.2 大模型的“认知层”：如何从文字意图提取水文 API 参数

如同我们在“双容水箱”和“氧化铝”项目中所做的那样，水务 Agent 的核心能力之一是降维。
当水务局长在对话框里输入：“看看昨天那场暴雨为什么会导致南城内涝，是谁的泵没开？”
`analyzer` 会自动将其拆解为多个 API 请求：
1. 提取“昨天”和“南城”的时空参数。
2. 调用 `get_basin_resource` 和 `run_sim` 查询历史回放数据。
3. 调取南城排涝泵站的实际动作记录。
4. 将对比结果汇编为结论。

## 10.3 人机协同（Human-in-the-loop）与安全护栏（Guardrails）机制

在 L4 级自主运行体系完全成熟之前，重大决策必须有“人机协同”。
`tests/hydrology/test_ch10_agent_role_boundary.py` 通过代码证实了各类 Agent 的边界。更重要的是，在发出如“开启泄洪闸”这样的灾难性动作之前，系统会强制触发护栏拦截，把最终的批准权通过 HydroDesktop 推送给值班长，实现了“机器算路，人类拍板”。

## 本章小结与证据锚点

- **核心结论**：多 Agent 分工协作与人机协同，是实现高度智能同时保障系统底线安全的必由之路。
- **代码锚点**：`core/hydrology/agents/dispatcher.py`, `core/hydrology/agents/analyzer.py`, `core/hydrology/agents/controller.py`
- **测试证据**：`tests/hydrology/test_ch10_agent_role_boundary.py`
