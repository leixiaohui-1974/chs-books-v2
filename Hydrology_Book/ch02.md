# 第2章 数字水文核心概念

## 2.1 流域、产汇流、入渗、基流与地下水-地表水交互

数字水文首先要把“水从哪里来、往哪里去、何时到达”讲清楚。
流域是最基本计算单元；降雨落到流域后，经过截留、入渗、地表产流、壤中流和地下径流等过程，最终汇入河网并形成出口流量。

核心水量平衡可写为：
$$\frac{dS}{dt} = P - ET - Q - G$$
其中：

1. $S$：流域蓄水量（地表、土壤、浅层地下综合）
2. $P$：降雨输入
3. $ET$：蒸散发损失
4. $Q$：地表与河道出流
5. $G$：深层渗漏或地下交换项

在过程分解上，常用：
$$Q = Q_s + Q_b$$
$Q_s$ 为快速响应径流，$Q_b$ 为基流（慢响应）。
地下水-地表水交互决定枯水期韧性，也影响洪峰消退形态。

`core/hydrology/concepts.py` 的核心逻辑，就是将这些概念从“术语解释”变成“可计算对象关系”：

1. 明确各变量的物理含义、单位和时间尺度。
2. 明确过程顺序与依赖关系（降雨->入渗->产流->汇流）。
3. 明确守恒约束与合法取值边界（如非负、上限、闭合误差）。

这使后续模型调用时不会因概念歧义导致输入错配或结果不可比。

## 2.2 场次洪水模型 vs. 连续水文模型

场次洪水模型（Event-based）和连续模型（Continuous）不是谁替代谁，而是面向不同任务边界的两类工具。

场次模型特点：

1. 聚焦单次降雨-洪水事件。
2. 参数少、部署快、适合应急研判。
3. 对初始土壤湿度和前期条件敏感。

连续模型特点：

1. 按连续时间更新土壤水、地下水和河道状态。
2. 适合长期调度、供需平衡和风险滚动评估。
3. 对数据质量与模型维护要求更高。

在工程中常见组合策略是：
连续模型提供“背景状态”，场次模型提供“事件强化分析”，二者耦合形成更稳健的预报链。

`core/hydrology/concepts.py` 在本节应体现的核心逻辑包括：

1. 模型适用域定义：什么问题用场次，什么问题用连续。
2. 状态初始化规则：事件模型如何继承连续状态。
3. 结果可比规则：统一指标体系（洪峰、峰现时、总量误差、告警命中率）。

因此，第2章不是做模型选型广告，而是建立“模型与任务匹配”的概念合同，为第3章核心水文能力实现打基础。

## 2.3 站点观测、网格化气象与遥感数据的融合

数字水文的难点往往不在“缺算法”，而在“多源数据不一致”：

1. 站点数据：精度高但空间稀疏。
2. 网格气象：空间连续但存在偏差和时效差异。
3. 遥感数据：覆盖广但受反演误差、云雨遮挡和重访周期影响。

融合目标不是简单求平均，而是构建“时空一致、语义一致、质量可追溯”的输入场。
典型融合流程包括：

1. 时间对齐：统一时区、步长、时间窗。
2. 空间对齐：点-面转换、网格重采样、流域汇总。
3. 质量控制：缺测补齐、异常剔除、质量标识（Quality Flags）。
4. 同化/校正：用可信观测约束模型状态或偏差场。

在代码架构上，第2章映射到 `src/hydrology/concepts.py` 与 `hydro_entities.py`，其中 `core/hydrology/concepts.py` 负责概念主干，`hydro_entities.py` 负责实体与字段约束。
`hydrology_server.validate_hydro_entities` 则把“概念正确”落实为“结构可校验”。

对应通过测试可引用 `tests/hydrology/test_ch02_entities_schema.py`，用于验证：

1. 实体字段完整性与类型一致性。
2. 单位、枚举、质量标识等约束有效。
3. 跨实体引用关系（如 Basin-Reach-Station）可闭合。

第2章完成了“数字水文概念层”的工程化定义：

1. 水文过程要回到守恒与过程链本体。

4. `concepts.py` 与实体校验测试共同构成后续算法章节的“概念地基”。
