# 第7章 面向关键水务系统的 API 设计

## 7.1 为什么水利系统需要 API-First 架构？

长期以来，水利信息化建设多以“项目制”为主，前端大屏与后端数据库深度绑定。这种紧耦合模式导致一旦有了新算法，或需要对接外部平台时，重构成本极高。
API-First（接口优先）架构彻底改变了这种局面。我们不再先画 UI，而是先设计系统的交互契约。在水文智能领域，API 是连接物理引擎（如 `rainfall_runoff.py`）与上层应用（如大模型、预警大屏）的唯一合法通道。

## 7.2 RESTful 资源模型：`/basins`, `/stations`, `/forecasts`, `/scenarios`

在 `core/hydrology/api/resources.py` 中，我们将错综复杂的水务实体全部微服务化为标准的 RESTful 资源。
- 当你需要查询某个流域的背景信息时，请求 `/basins/{id}` 即可返回一个规整的 JSON（即 `tests/hydrology/test_ch07_api_resource_contract.py` 验证的 `basin` 对象）。
- 预报结果作为一种资源挂载在 `/forecasts` 下，场景推演挂载在 `/scenarios` 下。
这种极简、标准的设计让外部开发者甚至非水务专业的 IT 团队，也能在几天内基于此搭建出水网监控应用。

## 7.3 时空窗口查询与异步任务（Asynchronous Jobs）处理

水文预报或 2D 水动力学仿真往往需要数分钟乃至数小时的算力开销。
如果采用同步 API，请求会面临超时失败。因此在 `core/hydrology/api/jobs.py` 中，我们引入了异步任务队列机制。
用户提交一个 `run_sim` 任务后，系统会立即返回一个 `job_id`，状态为 `pending`。上层系统可以根据该 ID 轮询进度，完成后再去统一获取海量结果集。
`tests/hydrology/test_ch07_async_jobs.py` 是这种长耗时调度模式在工程上落地的测试凭证。

## 本章小结与证据锚点

- **核心结论**：接口优先与异步任务架构，是支撑水务平台走向微服务与云原生时代的必由之路。
- **代码锚点**：`core/hydrology/api/resources.py`, `core/hydrology/api/jobs.py`
- **测试证据**：`tests/hydrology/test_ch07_api_resource_contract.py`, `tests/hydrology/test_ch07_async_jobs.py`
