## P0问题（必须立即修复）

- [ ] **P0-1: WNAL→WSAL 全章术语错误**。全章使用"WNAL"（Water Network Autonomy Level?），但CHS体系中的标准术语是"WSAL"（Water Systems Autonomy Level），见P1a §7.4、P2A及CLAUDE.md术语表§H。此错误出现约30+次，必须全局替换。这是体系一致性的根本性错误。

- [ ] **P0-2: §11.16图号与前文冲突**。§11.16中三张图编号为"图11-1"、"图11-2"、"图11-3"，与§11.2已有的"图11-1: 安全包络三区"严重冲突。应重新编号为图11-6、图11-7、图11-8（或按实际顺序调整）。

- [ ] **P0-3: §11.16图片路径不一致**。§11.1–11.15的图片路径为`./H/fig_11_XX_name.png`，但§11.16的三张图路径为`./figures/ch11_cbf_*.png`。必须统一为`./H/`路径格式。

- [ ] **P0-4: §11.1.2原理编号缺乏精确映射**。文中提及"原理一/二/三/四/五/八"，但未给出P1-P8的正式编号对应。根据CHS八原理体系（P1反馈、P2解耦、P3降阶、P4涌现、P5鲁棒、P6协调、P7适应、P8层级），安全包络应对应哪条原理？文中说"原理四（安全包络——先安全，后最优）"，但P1a中P4是Emergence（涌现），并非安全包络。需要明确安全包络在八原理中的精确位置，或说明第七章对原理的重新编排。

- [ ] **P0-5: 本章小结引用不存在的内容**。§11.15提到"控制屏障函数（CBF）为安全包络提供了严格的数学基础，CBF-QP控制器实现了性能优化与安全保证的统一（§11.16）"和"水利行业需借鉴航空（DO-178C）...（§11.17）"，但这两节出现在小结之后。结构上，小结应在所有正文内容之后。

## P1问题（重要但非阻塞）

- [ ] **P1-1: 内容严重冗余**。§11.2与§11.6内容高度重叠（三区划分定义、阈值标定方法重复讲述）；§11.3/§11.4与§11.7内容高度重叠（MIL/SIL/HIL对比表出现三次：表11-2、表11-7、表11-10）；§11.5.4的WNAL等级表（表11-8）与§11.9（表11-11）和§11.10（表11-12）高度重复。建议合并，全章可压缩30-40%。

- [ ] **P1-2: 三个工程实例结构雷同且数据空洞**。§11.8（水电站）、§11.11（灌区）、§11.12（梯级电站）三个案例的"核心挑战"完全相同（逐字复制："安全约束复杂...控制策略验证困难...人机职责边界模糊"）。§11.12的"实施效果"全是"显著降低/提升/缩短"等定性描述，无任何定量数据。建议保留1-2个精品案例，给出具体数据。

- [ ] **P1-3: CBF部分（§11.16）符号与CHS体系不完全一致**。CBF中使用$h(\mathbf{x})$表示屏障函数，但CHS体系中$h$已用于水位偏差（见CLAUDE.md §I符号表）。建议CBF屏障函数使用$b(\mathbf{x})$或$B(\mathbf{x})$，避免符号冲突。

- [ ] **P1-4: 式(11-1)三区定义与§11.6.1不一致**。§11.1.3定义绿区为$[x_{\min}+\delta, x_{\max}-\delta]$，但§11.6.1定义绿区为"水位±10cm"，黄区为"±10-20cm"。两种定义方式不同且数值不对应，读者会困惑。

- [ ] **P1-5: 僵尸引用**。以下文献在参考文献列表中出现但正文未引用：[11-17] Koopman & Wagner 2017、[11-18] Kalra & Paddock 2016、[11-19] Shneiderman 2022、[11-20] Negenborn & Maestre 2014、[11-21] Maestre & Negenborn 2014。应在正文中适当位置引用或从列表中删除。

- [ ] **P1-6: 自引率偏高**。参考文献共23条，其中Lei/雷晓辉自引5条（[11-8][11-9][11-22][11-23]及隐含的P1a引用），占比约22%。[11-22][11-23]也是僵尸引用（列出但正文未引用）。建议补充更多第三方文献（如Litrico & Fromion的渠道控制相关工作、Clemmens等人的灌溉自动化工作）。

- [ ] **P1-7: §11.16.7 CBF案例中管道弹性容量符号与P1a不一致**。文中定义$C_h = 100$ m·s²，但P1a中管道弹性容量$C_h$的单位应为m·s²（$C_h = \rho g A L / a^2$），且数值100的工程依据未说明。需给出管道参数（长度、直径、波速）以支撑此数值。

- [ ] **P1-8: 场景覆盖率公式（§11.4.1）理论基础薄弱**。$C_{\text{total}} = w_1 C_{\text{state}} + w_2 C_{\text{boundary}} + w_3 C_{\text{fault}}$的权重$w_1=0.4, w_2=0.3, w_3=0.3$缺乏理论依据，仅注"可根据工程特点调整"。建议说明权重选择的工程逻辑或引用相关标准。

## P2问题（建议改进）

- [ ] **P2-1: 引导案例缺少回溯**。开篇胶东调水案例生动，但后文未再引用。建议在§11.2（三区划分）或§11.4（在环验证）中回溯此案例，说明其安全包络参数是如何标定和验证的。

- [ ] **P2-2: 表格编号不连续**。表11-1到表11-12，但实际只有约10张表，部分编号跳跃。建议通读检查编号连续性。

- [ ] **P2-3: §11.16.4 HOCBF定义中递归公式不够严谨**。$\psi_1(\mathbf{x}) = \dot{\psi}_0 + \alpha_1(\psi_0)$中，$\dot{\psi}_0$依赖于控制输入$\mathbf{u}$，但定义中未明确区分。标准HOCBF定义（Xiao & Belta 2019）中使用的是$L_f\psi_0$而非$\dot{\psi}_0$。

- [ ] **P2-4: §11.5.1类比可以更精确**。MIL类比为"图上推演"稍显不当（MIL是定量仿真而非定性推演）。建议改为"风洞试验"或"计算机仿真"。

- [ ] **P2-5: §11.16.2中$q_{in}$与CHS符号体系的关系**。CHS中使用$Q$表示绝对流量、$q$表示偏差量（CLAUDE.md §I），但CBF案例中混用$q_{in}, q_{out}$作为绝对量和$Q_{pump}$作为绝对量，不一致。

- [ ] **P2-6: 练习题第11题中状态空间覆盖率计算有歧义**。"4个变量×10个区间"的总网格点数应为$10^4 = 10000$，但题目未明确是组合还是独立。建议明确说明。

- [ ] **P2-7: §11.13平台建设指南与§11.7高度重复**。建议合并为一节，或将§11.13定位为"清单与模板"，删除重复的流程描述。

## 亮点

- **引导案例设计出色**：胶东泵站案例和§11.2.6反面案例（供水管网事故）都非常具体、有说服力，能够有效传达安全包络的工程价值。

- **CBF理论引入得当（§11.16）**：将控制屏障函数与三区安全包络建立数学联系，为"先安全后最优"提供了严格的理论基础。CBF-QP的"最小侵入性"定理与安全包络的设计哲学完美契合。泵站CBF案例（§11.16.7）给出了完整的从建模到仿真的闭环示例。

- **国际标准对比（§11.17）**：系统对比ISO 26262、DO-178C、ISO 21448、IEC 61508，并分析水利行业五个特殊性，对行业标准化建设有实际指导价值。

- **练习题分层设计好**：L1-L5分层覆盖从基础理解到计算题到开放讨论，第10-11题的定量计算尤其好，学生可以动手练习场景覆盖率和安全裕度计算。

- **约束优先级层次清晰**：结构安全 > 服务安全 > 效率约束的三级优先级定义简洁明确，工程实用性强。

---

**总体判定：Major Revision**

本章覆盖面广、案例丰富，但存在两个根本问题：(1) WNAL→WSAL术语错误贯穿全章；(2) 内容冗余严重（三区定义重复3次、MIL/SIL/HIL对比表重复3次、WSAL等级映射表重复3次）。修订重点应放在：统一术语、合并重复内容（压缩30-40%）、统一图号和路径、充实工程案例数据。CBF理论部分质量较高，但需修正符号冲突和图号问题。

## P1问题（重要但非阻塞）
- [ ] **三区阈值没有真正接入 CHS 参数体系。** 当前只给 `δ=kσ` 或 `δ≥v_maxT_response`，没有把测量时延 `τ_m`、通信/执行时延 `τ_d`、执行器动作时间、模型误差预算纳入。建议改为类似 `δ ≥ v_max(τ_m+τ_d+τ_exec)+ε_model`，明渠再叠加传递函数峰值响应。
- [ ] **§11.2.2 的“80%-90% / 95%-100%”经验阈值过粗，且与前文 δ 法冲突。** 对渠道、水库、泵站、管网，这种按百分比切阈值都不稳妥。建议改成“按工程类型分表给法”，至少分明渠、压力管网、泵机设备三类。
- [ ] **黄区/红区控制策略写得太理想化。** “黄区缩短预测时域”“一律降级为 PID”“红区 MRC+人工接管”都不是通用工程答案。长时滞渠道缩短预测时域可能更危险；很多场景红区应由 PLC/联锁直接进保护，不是上位机再切 MRC。建议改成“收紧约束、限制控制增量、冻结上层优化、必要时切本地规则控制/联锁”。
- [ ] **SIL/HIL 工具表有工程事实问题。** `dSPACE ControlDesk`、`NI VeriStand` 更像测试编排/HIL 环境，不宜当成典型 SIL 核心工具；`Raspberry Pi + Simulink Real-Time` 这条基本不成立。建议按“虚拟 PLC/目标代码主机执行/实时仿真平台/网络仿真器”重写。
- [ ] **价格、周期、覆盖率等数字缺依据。** 如“MATLAB 50 万/年”“HIL 12-18 个月”“覆盖率≥95%”“求解 1.2 ms”“72 小时”“验证周期缩短到 3 天”等都需要注明来源、年份、适用规模；否则像投标宣介，不像教材。
