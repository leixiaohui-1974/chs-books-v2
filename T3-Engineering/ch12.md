# 第 12 章：产业链责任矩阵与事故状态空间溯源（RACI）

## 引言

在《水网智能化标准体系》被确立为行业法规之前，水利信息化领域存在一个难以言说的"潜规则"：系统平时正常运转时，皆大欢喜；一旦遇到复杂工况导致水位失控、漫堤或泵机烧毁，事故追责往往陷入一团乱麻。

设计院会说："我们的水力学计算是保守的，是软件厂商乱发指令。"软件厂商会反驳："我们的代码逻辑没问题，是传感器数据漂移误导了算法，或者底层 PLC 的死区太大。"而最终，这口黑锅往往由当时正在值班的、对代码黑盒一无所知的调度员来背。

这种相互推诿的根源，在于**缺乏一张基于状态空间和控制权的法定责任矩阵**。水网智能化本质上是一次控制权的再分配（从人到机器），如果责任边界不随之进行严密的"数学解耦"，高阶智能系统将成为无人敢用的"烫手山芋"。

从控制论视角看，责任分配问题是控制权解耦问题的治理映射。《水系统控制论》[1]（T1-CN）第三章所述的"分层分布式控制"原理将物理系统的控制权分配到不同层级；《智能与自主》（T2b）第八章 WNAL 评估框架则明确了人—机控制权在各等级之间的法定边界。本章将这些控制论概念转化为可执行的产业链责任矩阵。

本章将利用控制论的解耦思维，构建一张清晰的**RACI（谁负责、谁执行、谁咨询、谁知情）产业链责任矩阵**，提供高频审计日志的最低要求规范，以决策因果树为例还原典型事故的责任链，并阐述调度员接管能力认证体系与 AI 自主时代的责任归属框架演进。

## 12.1 产业链控制权与责任解耦（RACI 矩阵完整版）

在 L3 及以上的自治系统中，"系统"不再是一个单一供应商交付的软件，而是由设计院的动力学模型、科研院所的调度算法、集成商的边缘网关以及硬件厂商的 PLC 和传感器共同组成的"共生体"。每一方都必须认领其在"状态空间转移"中的法定责任。

RACI 矩阵中，角色定义如下：

- **R（Responsible，执行责任）**：直接执行该决策点工作的主体
- **A（Accountable，问责责任）**：对该决策点最终结果负连带法律责任的主体，每个决策点最多一个 A
- **C（Consulted，被咨询方）**：须征询意见但不负主责的专业参与方
- **I（Informed，知情方）**：须告知但无须深度介入的利益相关方

### 12.1.1 设计院的责任：物理包络与理论可控性

在传统的土建工程中，大坝垮了找设计院。在水网智能化中，这一逻辑被升维：

- **核心交付**：水动力学模型（Plant Model）、ODD（运行设计域）的数学边界参数（D01）、MRC（最小风险状态）的物理安全包络（D02）。
- **法定责任**：设计院必须对系统的**理论可控性**负责。如果在发生漫堤事故后，溯源发现事故时的流量和气象状态**处于设计院声明的 ODD 范围之内**，且物理设施本身未能承受住模型计算出的最恶劣水锤效应，则这是"设计缺陷"，设计院承担首要责任（对应 RACI 矩阵中 D01、D02 的 A 项）。

### 12.1.2 算法/模型商的责任：控制偏差与算法收敛

- **核心交付**：封装为 FMI 标准 [7] 的控制律模型（Controller FMU）或 AI 调度算法（D04）。
- **法定责任**：算法商对系统在 ODD 范围内的**状态收敛性**负责。如果系统状态在 ODD 边界内，传感器输入正常，但算法却输出了发散的指令（如在平稳工况下频繁正反转闸门，引发谐振），导致设备物理损坏，则算法供应商承担首要责任。

### 12.1.3 集成商的责任：时序确定性与逻辑保真度

- **核心交付**：边缘控制网关、网络通信链路、底层 PLC 程序，以及高频数据记录黑匣子。
- **法定责任**：集成商不对算法的"聪明程度"负责，但必须对**指令的忠实传递与时限保证**负责。如果在事故中，算法发出了正确的 MRC 降级指令，但由于边缘网关的性能卡顿（时延超标）或代码死锁导致指令未能下达给 PLC，或者底层 PLC 在接收到越限指令时未能行使"物理拒执权"，则集成商承担首要责任。

### 12.1.4 建设方与运维单位的责任：日常审计与接管响应

- **核心交付**：提供真实的运行环境、调度员，以及维持传感器准确度的运维经费。
- **法定责任**：当系统因不可抗力或硬件老化跌出 ODD 边界，并向控制中心发出"接管请求"时，运维单位的调度员必须在规定的响应时间（如 5 分钟）内完成接管。如果系统成功触发了 MRC，但因为调度员脱岗未能接管导致的次生灾害，由运维方担责（对应 D14）。

### 12.1.5 认证机构（CAB）的责任：独立见证与准入背书

- **核心职责**：第三方合格评定机构对 HIL 测试结论负独立见证责任（D07、D09），对事故根因分析提供中立裁定意见（D18）。
- **法定责任**：CAB 机构若因测试程序失当（如接受虚假日志）而出具了失实的准入报告，须对因此发生的事故承担专业过失的连带责任。

## 12.2 审计日志的最低要求：高频黑匣子规范

RACI 矩阵确定了"谁该负责"，而**审计日志**则是"如何证明谁负责"的物理证据。没有可靠日志，再严密的责任矩阵也只是纸上谈兵。

### 12.2.1 哪些操作必须有审计日志

标准体系规定，以下操作必须强制记录不可篡改的审计日志，任何未记录的操作在事故调查中推定为"不利于该责任方"：

**表 12-2 高频审计日志强制记录项目清单**

| 类别 | 必记事件 | 最低记录频率 | 最低保留期 |
|------|---------|------------|----------|
| 传感器数据 | 水位、流量、压力等核心过程量 | 1 Hz（每秒 1 次） | 5 年 |
| 算法决策 | MPC 优化输出（每次调度的目标函数值、约束是否活跃、求解时间） | 每次调度周期 | 5 年 |
| 执行器指令 | 向 PLC 发出的每一条闸门/泵站指令（含时间戳、指令值、来源模块） | 每次下发即记 | 5 年 |
| 执行器反馈 | PLC 对指令的确认回执（含实际执行值、执行延迟） | 每次回执即记 | 5 年 |
| ODD 监测 | ODD 边界监测器的状态输出（各维度的当前值、告警等级） | 1 Hz | 5 年 |
| 状态机转移 | 四态机（正常/受限/降级/接管）每次状态切换的触发条件与时间 | 每次转移即记 | 10 年 |
| 人工操作 | 调度员的所有手动干预（覆盖 AI 指令、参数调整、接管确认等） | 每次操作即记 | 10 年 |
| 系统健康 | 边缘网关 CPU 占用率、内存使用率、网络延迟（毫秒级） | 1 Hz | 1 年 |
| 代码变更 | OTA 升级的部署时间戳、版本号、操作人身份 | 每次变更即记 | 永久 |
| 模型参数 | FMU 模型参数的在线调整记录（含修改前后的值及授权人） | 每次修改即记 | 永久 |

### 12.2.2 日志格式的技术要求

为确保日志可被独立机构读取和核验，格式要求如下：

1. **时间同步**：所有节点（云端服务器、边缘网关、PLC）的时钟须通过 NTP/PTP 协议同步，精度 $\le 1$ ms。日志中的时间戳采用 UTC 标准时间，精确至毫秒。
2. **不可篡改性**：日志文件须采用"追加写入（Append-Only）"模式，每条记录附带前一条记录的 SHA-256 哈希值，形成哈希链。任何对历史日志的篡改均可被检测。
3. **结构化存储**：采用 JSON-Lines 或 Apache Parquet 格式存储，字段命名规范须公开发布，确保第三方可独立解析。
4. **离线备份**：日志须在本地存储的同时，每日向监管机构指定的独立存储节点进行加密备份，确保即使系统遭受恶意攻击或硬件损毁，日志依然可以被恢复。
5. **完整性证明**：每 24 小时生成一次日志摘要文件（含当日所有日志的 Merkle 树根哈希），由运维单位和 CAB 机构各留存一份，用于事后核验。

## 12.3 决策因果树：以"算法发散导致水位超限"事故为例

当事故发生时，RACI 矩阵给出了责任认领的框架，高频日志提供了证据原材料，而**决策因果树（Causal Tree）**则是将两者连接起来的分析工具——它将散乱的日志时间序列，整理为一棵清晰的"原因-结果"树状结构，直观呈现事故的责任链。

### 12.3.1 事故背景描述

某明渠调水工程（WNAL L3 级，部署 MPC 全自动调度）在某日 14:32 发生水位超限事故：2 号渠段水位超过堤顶警戒线 $0.45$ m，触发就地溢流保护，并造成两侧农田被淹，经济损失约 80 万元。事故发生时，调度员正在进行值班交接。

### 12.3.2 因果树构建过程

调查组从日志中提取事故前 1 小时的数据，构建如下因果树：

```
[顶层事件] 2号渠段水位超堤顶警戒线 0.45 m（14:32:17）
    |
    +—— [直接原因A] 节制闸 CH-04 开度在 14:28:00~14:32:00 期间持续增大至 100%
    |       |
    |       +—— [原因A1] MPC 算法在 14:25:00 的调度周期输出 CH-04 目标开度 = 100%
    |       |       |
    |       |       +—— [原因A1a] 算法输入中，上游流量传感器 FT-02 示值为 92 m³/s
    |       |       |            （实际来流仅 41 m³/s，传感器零点漂移导致虚假数据）
    |       |       +—— [原因A1b] MPC 模型认为下游需水量大，需加大过流，指令合理
    |       |            [结论：算法本身逻辑正确，输入数据失真]
    |       |
    |       +—— [原因A2] 底层 PLC 接收到"开度 100%"指令后全量执行，未触发物理限位
    |               |
    |               +—— [原因A2a] PLC 安全逻辑中，对 2 号渠段上游水位已接近警戒线
    |                            时的"减速开限"保护未正确配置（设定值为 堤顶 + 0.1m，
    |                            而非堤顶 - 0.3m）
    |                    [结论：PLC 安全拦截逻辑配置存在设计错误]
    |
    +—— [直接原因B] ODD 监测器未发出传感器异常告警
    |       |
    |       +—— [原因B1] FT-02 传感器的量程为 0~100 m³/s，漂移后示值 92 m³/s 仍在量程内
    |               |
    |               +—— [原因B1a] ODD 监测器仅设置了量程越限告警，未设置基于历史
    |                            数据的统计异常检测（如与相邻时段均值偏差 > 3σ 告警）
    |                    [结论：ODD 监测器设计不完善，未覆盖渐变型传感器漂移]
    |
    +—— [间接原因C] 调度员值班交接期间无人监控大屏，错过 14:28 的水位上升趋势
            |
            +—— [原因C1] 运维规程中"交接班期间双岗值守"要求未被执行
                    [结论：运维单位操作规程执行管理缺位]
```

### 12.3.3 责任认定结论

依据因果树分析与 RACI 矩阵对照：

| 责任方 | 责任事项 | 责任性质 |
|--------|---------|---------|
| **集成商** | PLC 安全拦截逻辑配置错误（原因 A2a），属于 D06/D07 阶段 SIL/HIL 验证漏洞 | 首要责任 |
| **集成商** | ODD 监测器未覆盖渐变型传感器漂移（原因 B1a），属于 ODD 监测器代码设计不完善 | 连带责任 |
| **硬件供应商** | FT-02 传感器发生零点漂移，属于设备质量与标定周期管理问题（D19） | 次要责任 |
| **运维单位** | 交接班双岗制度执行缺位（原因 C1），属于管理责任（D14） | 次要责任 |
| **算法/模型商** | 算法本身逻辑正确，输入数据失真导致的决策偏差不由算法方承担 | 免责 |
| **设计院** | ODD 定义中对传感器精度的要求（$< 1\%$ FS，见 D01）已涵盖此类漂移，属合规设计 | 免责 |

## 12.4 事故追溯标准：基于高频日志的状态空间断案

当严重的物理溃坝、漫堤或断流事故真实发生时，调查组不再需要听取各方的雄辩，而是直接切入系统独立运转的"黑匣子"——**高频决策逻辑归因日志（Causal Tree Log）**，通过回溯事故前一小时的系统状态轨迹来进行"控制论断案"。

### 12.4.1 断案第一步：确认事故时的边界条件（ODD 检验）

调查组调取事故发生前时刻的状态向量 $x(t_0)$ 和扰动向量 $d(t_0)$：

- **结论 A**：如果此时遭遇了未声明的超标洪水（越出 ODD），系统免责。责任转入第二步：系统是否正确执行了退出机制？
- **结论 B**：如果状态依然在 ODD 内部，说明这是系统声称"有能力处理"的工况。事故责任必定在控制系统的设计或代码层，进入第三步排查。

### 12.4.2 断案第二步：越界工况下的降级检验（MRC 检验）

针对结论 A（已越出 ODD）：

- 如果系统没有报警并继续盲目下发常规优化指令，导致灾难扩大，判定为**集成商 ODD 监测器代码失效**；
- 如果系统报警并尝试降级，但触发了依赖云端通信的死亡逻辑，判定为**设计院 MRC 架构设计严重违规**；
- 如果系统成功退回到了本地 PLC 设定的 MRC 保底状态，但大水依然漫过了堤坝，判定为**不可抗力的天灾，系统各方完全免责**。

### 12.4.3 断案第三步：包络内失控的控制律分解（算法与硬件对质）

针对结论 B（系统在 ODD 内却未能有效控制）：调查组在仿真平台中，利用事故期间的实际传感器输入序列，重放在环仿真：

- **情况 1（算法幻觉）**：仿真中算法依然输出发散或剧烈震荡的指令，判定为**算法供应商模型训练过拟合或算法发散**；
- **情况 2（传感失效或通信硬伤）**：仿真中算法输出正常，但现场日志显示传感器漂移或网络延迟，判定为**硬件供应商或网络集成商责任**；
- **情况 3（物理联锁失效）**：算法发出危险指令但 PLC 未拦截，判定为**集成商未设置安全拒执权的渎职责任**。

## 12.5 接管能力认证：调度员认证考核的具体内容和频次

在 L3 级系统中，调度员的接管能力是整个安全体系的最后一道防线。然而，随着系统智能化程度提高，调度员日常接触操作的机会越来越少，其接管能力面临持续衰退的风险——这正是自动驾驶领域著名的"技能生锈（Skill Degradation）"问题。

### 12.5.1 认证考核的具体内容

调度员接管能力认证考核分为三个模块：

**模块一：理论知识考核（笔试，100 分，满分标准 $\ge 80$ 分）**

- 水网自主运行等级（WNAL）及各等级调度员职责（20 分）
- ODD 边界参数的含义与当前工程的具体数值（20 分）
- 四态机状态转移逻辑及接管请求的解读方法（20 分）
- 常见 P0/P1 场景的应急处置规程（20 分）
- 系统日志的基本阅读与关键指标判断（20 分）

**模块二：仿真实操考核（在培训模拟机上，时限 90 分钟）**

| 考核场景 | 分值 | 通过标准 |
|---------|------|---------|
| P0-01：全域通信中断，接管并维持水位稳定 | 25 | 在 $T_c$ 内完成接管，水位偏差 $< 0.3$ m |
| P0-02：核心传感器失真，识别异常并切换备用 | 25 | 识别时间 $< 3$ min，未因误操作扩大偏差 |
| P1-01：单泵站故障，重新分配调度方案 | 20 | 5 分钟内完成方案调整，全线水位满足目标 |
| P2-01：超标洪水预警，制定应急方案并上报 | 15 | 方案合理，上报格式符合规范 |
| 综合应急操作（随机抽取 1 个 P0 场景） | 15 | 无漫堤、断流等严重后果 |

实操总分 $\ge 70$ 分视为通过。

**模块三：心理状态与认知负荷评估**

在仿真实操考核过程中，同步监测调度员的操作延迟分布与决策路径，识别是否存在"认知休眠"迹象（如在场景初期反应时间超过 30 秒，或出现明显的误操作后长时间犹豫）。评估结论由心理行为专家参与出具，作为综合评定的参考项（不作为单独一票否决项，但会影响认证等级评定）。

### 12.5.2 认证频次与执照管理

- **初次认证**：上岗前完成，合格后发放"调度员接管认证证书（WNAL L3）"。
- **年度复训与考核**：每年一次，重点考核 P0 场景实操（随机抽取，不提前告知具体场景）；通过者续签证书，未通过者暂停上岗资格，限 60 天内重考一次。
- **事故后强制复训**：若调度员在真实事故中因操作不当（经事故调查确认）导致损失扩大，须完成为期 30 天的强制复训并重新认证。
- **季度演练（不记分）**：除年度考核外，运维单位须每季度组织一次全员仿真演练，演练记录存入个人培训档案；连续 2 个季度未参加演练的调度员，视同未完成年度考核处理。

## 12.6 科研模型向工程转化的"责任买断"机制

水利工程事关国计民生。在明确了严酷的追责标准后，许多科研院所可能会在承接智能化项目时产生退缩心理。为了鼓励科研成果转化，标准体系引入了**"责任买断（Liability Buyout）"**机制：

1. **明确交付物边界**：科研院所只负责交付符合 FMI 标准的算法模型（FMU），并附带严密的 ODD 数学推导证明。科研院所**不对代码在异构操作系统下的内存溢出、死锁、网络延迟等 IT 级错误负责**。
2. **集成商的"打包承保"**：专业的软件集成商或云厂商，在拿到科研院所的 FMU 后，通过严格的 SIL 和 HIL 测试，将其封装进自己的生产级工业网关或云平台，承担系统在线运行时的可用性（SLA）及代码稳定性风险。
3. **业主的责任赦免**：如果系统严格按照本标准通过了 CAB 的第三方见证测试并发放了准入包，那么在发生小概率的"黑天鹅"算法失控事故时，建设方只要没有违规操作，即可免于"盲目上马信息化项目"的行政追责。

## 12.7 未来趋势：AI 自主时代的责任归属框架演进

### 12.7.1 自动驾驶行业的经验借鉴

自动驾驶行业已经走在水网智能化行业的前面，在责任归属框架方面积累了大量制度经验。其演进历程对水网行业具有直接的参考价值：

**第一阶段（辅助驾驶，L1/L2）**：驾驶员始终是法律上的"操作责任主体"，自动驾驶系统的失效被视为"产品质量问题"，制造商承担产品责任。对应水网行业：WNAL L1/L2，调度员承担操作责任，集成商承担产品质量责任。

**第二阶段（条件自动化，L3）**：SAE J3016 [3] 明确规定，在 L3 系统声明的 ODD 范围内，系统是"操作责任主体"，驾驶员仅承担"接管响应"责任。若驾驶员在系统发出接管请求后的规定时间内未接管，驾驶员担责；若接管请求发出前系统已发生事故，制造商担责。这一框架与本章 RACI 矩阵的设计逻辑高度吻合。对应水网行业：WNAL L3 的 ODD 内事故由集成商/算法商担责，ODD 外接管响应迟误由运维方担责。

**第三阶段（高度自动化，L4/L5）**：责任框架正在向"产品责任"而非"驾驶员责任"彻底转移。德国 2021 年修订的《道路交通法》（StVG）明确规定，L4 级自动驾驶车辆的事故责任由制造商承担，驾驶员完全免责（只要其未进行非授权的手动干预）。与此同时，制造商须向政府机构提交"技术监督系统"，对车辆远程状态进行实时监控。

### 12.7.2 水网 AI 自主时代的责任框架展望

随着水网 WNAL 等级向 L4/L5 演进，现有 RACI 矩阵的若干假设将面临重构：

**挑战一：谁是"操作者"？**

当系统达到 L4 级（高度自主，无需人工接管）时，"调度员接管"这一概念将消失，RACI 矩阵中的 D14、D15 条目须重新设计。届时，算法供应商将成为大多数 ODD 内事故的唯一责任主体，其法律地位将类似于自动驾驶汽车的 OEM 制造商。

**挑战二：AI 决策的可解释性要求将成为法定义务**

当前，"黑盒决策无因果"仅是一票否决条款（第 10 章条款 9）。在 L4/L5 时代，可解释性将被写入产品责任法的条文——如果 AI 系统无法在事故发生后提供人类可理解的决策归因，推定为制造商过失，无需原告证明因果关系。

**挑战三：持续学习算法的责任归属**

当算法在运行期通过在线学习持续更新参数时（如强化学习策略），传统的"版本号"追溯逻辑将失效——算法在事故发生时的参数状态，与上次 HIL 测试时的参数状态可能存在数千次梯度更新的差距。对此，建议参考自动驾驶行业正在形成的"持续学习算法的安全认证框架"，要求算法的在线学习过程本身受到额外的安全约束（如禁止在高风险工况下学习更新），并将每次参数更新视为一次"Moderate OTA 变更"进行管控。

这套严密的 RACI 矩阵与断案机制，彻底打碎了水利信息化领域"出了事大家一起和稀泥"的混沌状态。它使得水网系统中的每一行代码、每一个参数、每一个网络节点，都背负上了沉甸甸的法律与工程责任，从而用制度的力量，逼迫整个系统走向真正的"安全与自治"。
